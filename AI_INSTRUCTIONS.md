# AI開発ルール・設計方針

## このファイルの目的

このファイルは、今後AIがこのリポジトリで作業するときに読む補助資料です。

**このファイルは補助情報です。実装前には必ず現在のコードを確認し、このファイルと実際のコードに矛盾がある場合は現在のコードを優先してください。**

## アプリの目的

- 小学1年生〜3年生向けの算数総復習Webアプリ。
- 子どもが紙を使わず、選択式問題・メモ・復習機能を使って学習できることを目指す。
- 計算、文章題、筆算、時計、単位、図形、表とグラフなどを扱う。
- 間違えた問題を保存し、苦手問題として再挑戦できる構成を目指す。
- 保護者が学習履歴や苦手分野を確認できることも目的に含む。

## 開発方針

- 既存機能を壊さないことを最優先にする。
- 子どもが使いやすいことを優先する。
- スマホ・タブレットでの操作性を重視する。
- 依頼範囲外の大規模リファクタリングはしない。
- 依頼外の改善は原則として実装せず、最後に提案としてまとめる。
- 必要な最小限の修正は行ってよい。
- 新しいライブラリを勝手に追加しない。追加が必要な場合は、理由を明確にする。
- 外部API、課金サービス、サーバー連携が必要な実装は勝手に進めない。
- localStorageの保存形式を変更する場合は、既存データとの互換性に注意する。
- 子ども向け文言は学年に合わせ、漢字には可能な範囲でふりがなを付ける。
- 問題は原則として選択式を維持する。
- 作業後は `npm run build` を確認する。
- 変更内容と確認結果を簡潔に報告する。

## 重要なファイル構成

- `package.json`
  - npm scriptsと依存関係を管理する。
- `src/main.jsx`
  - Reactアプリのエントリーポイント。
- `src/App.jsx`
  - 画面遷移、クイズ開始、結果処理、スタンプ・賞状判定などの中心。
- `src/data/questions.js`
  - 学年、単元説明、問題データを管理する。
  - 新しい単元や問題を追加する場合は優先して確認する。
- `src/components/`
  - 画面・UIコンポーネント群。
  - 主な画面: `Home.jsx`, `UnitSelect.jsx`, `QuizScreen.jsx`, `ResultScreen.jsx`, `ParentDashboard.jsx`
  - 主な表示部品: `ClockFace.jsx`, `ShapeDisplay.jsx`, `GraphDisplay.jsx`, `VerticalWork.jsx`, `MemoBoard.jsx`, `RubyText.jsx`
- `src/utils/`
  - 保存、統計、問題抽出、効果音、スタンプ、賞状のロジック。
  - 主なファイル: `storage.js`, `quizUtils.js`, `stats.js`, `sound.js`, `stamps.js`, `awards.js`
- `src/styles.css`
  - 全体スタイル、レスポンシブ、アニメーション、時計・図形・グラフ表示など。

## 壊してはいけないもの

コードから確認できる範囲:

- `npm run build` が成功すること。
- ホーム画面から主要画面へ遷移できること。
- 学年選択と単元選択。
- サクサク10問モード。
- じっくり5問モード。
- クイズ出題、回答、正誤判定、結果表示。
- 文章題のステップ式出題。
- 筆算のステップ式表示。
- `RubyText` によるふりがな表示。
- localStorageへの保存処理。
- 苦手問題の保存・復習。
- 保護者画面。
- メモボード。
- 効果音オン・オフ。
- スタンプ帳。
- 賞状機能。
- 時計、単位、図形、表とグラフ問題。
- スマホ幅で大きく破綻しないレスポンシブ表示。

要確認:

- スマホ実機でのタッチ操作。
- localStorageの既存データとの完全な互換性。
- ふりがなの網羅性。
- 全単元・全問題の正答判定。
- スタンプ・賞状の条件が現在の単元名と完全に一致しているか。

## Codexが作業するときの手順

1. `AI_INSTRUCTIONS.md` を読む。
2. `PROJECT_STATUS.md` を読む。
3. 依頼内容に関係するコードを確認する。
4. 影響範囲を小さくして修正する。
5. 依頼外の変更をしない。
6. `npm run build` を実行する。
7. 変更内容・確認結果・注意点を報告する。

## 作業後の報告ルール

作業後は、以下を簡潔に報告する。

- 変更したファイル
- 変更内容
- 動作確認方法
- `npm run build` の結果
- 注意点
- 次にやるとよさそうなこと

## ドキュメント更新ルール

- 大きな機能追加をした場合は、必要に応じて `PROJECT_STATUS.md` を更新する。
- 開発ルールや設計方針が変わった場合のみ、このファイルを更新する。
- ドキュメント更新だけの依頼では、コードを変更しない。
